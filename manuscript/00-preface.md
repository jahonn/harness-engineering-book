# 前言

软件工程正在进入一个新的中间层时代。

过去几年，大模型的讨论常常集中在模型本身：参数规模、上下文窗口、推理能力、多模态输入、代码生成能力、基准分数和价格。这样的关注是合理的。没有足够强的模型，就不会有今天意义上的智能体系统。但是，模型进入工作环境后，另一个事实会迅速浮现：模型能力并不会自动变成可靠的软件行为。

一个模型可以写出看似合理的命令，却不知道当前目录是否正确；可以生成补丁，却不了解用户未提交的修改；可以解释测试失败，却在没有权限边界的情况下执行危险操作；可以把工具调用组织成多步计划，却在上下文压缩后忘记最关键的约束；可以在演示中完成任务，却在真实仓库、真实依赖、真实审批、真实成本和真实事故责任面前暴露系统性脆弱。

这类问题不能只靠提示词解决，也不能只靠更换模型解决。它们属于 Harness Engineering 的范围。

本书把 harness 定义为模型与真实任务环境之间的运行基底。它负责把用户目标、系统指令、项目规则、工具、文件系统、外部服务、权限、记忆、评测、可观测性和人工协作组织成一个可运行的系统。模型负责生成候选行动，harness 负责限定这些行动在什么条件下可以发生、发生后如何被观察、失败时如何恢复、经验如何沉淀为下一次更好的行为。

提示词工程（prompt engineering）关注的是：怎样让模型在一次或几次调用中说对。Harness engineering 关注的是：怎样让一个由模型驱动的系统在真实环境中持续做对，并且在做错时可控、可查、可改。两者的分界不在修辞，而在工程责任。

本书的写作对象不是模型训练研究者，也不是只想复制几个提示词模板的工具使用者。它面向正在把智能体系统带入生产环境的人：coding agent 产品的工程师，企业内部 agent 平台的负责人，研发效能团队，基础设施团队，安全治理团队，以及希望理解智能体系统真实复杂度的技术管理者。

本书会保留一部分英文工程术语。在智能体系统中，harness、agent、trace、eval、profile、checkpoint 等词已经成为运行时对象、证据对象或配置对象，强行翻译会损失边界。书中会在概念首次出现时给出中文解释，之后按固定写法使用；已有稳定中文表达的词，如上下文、状态、权限、审批、工具、评测、可观测性、回滚和复盘，则优先使用中文。

书中会反复强调一个判断：智能体可靠性主要来自运行基底的系统设计，而不是某一次完美提示或某个单点框架。工具 schema、权限策略、工作区隔离、上下文装配、长期记忆、trace、checkpoint、评测任务、人工审批、终端界面、插件协议、成本控制和持续演化机制，都是同一个问题的不同侧面。

按这个口径，本书不会把任何一个产品或框架当作标准答案。Claude Code、OpenAI Codex、MCP、OpenAI Agents SDK、SWE-bench、Terminal-Bench，以及匿名工程案例，都只会作为材料和案例进入讨论。本书关心的是这些系统背后的工程结构：它们如何把模型能力接入文件系统，如何决定工具能否执行，如何在任务失败后保留证据，如何把一次人工纠错沉淀为长期规则，如何让一个看似“会写代码”的模型进入一个有边界、有审计、有回滚的软件工程过程。

本书也不会回避限制。智能体系统会误判、遗忘、过度自信，也会在复杂任务中走偏，把局部正确当作全局完成。成熟 harness 不假设模型永远正确；它把“模型可能错”作为架构前提。权限、观测、评测、恢复和人机协作由此成为核心结构。

写作上，本书采用从概念到结构、从运行到治理、从评测到演化、最后进入案例和成熟度模型的顺序。第一编解释为什么需要 Harness Engineering；第二编拆解 harness 的核心结构；第三编讨论安全、权限和治理；第四编进入可观测性与评测；第五编讨论产品化 Agent OS；第六编讨论持续演化；第七编给出案例和成熟度模型。

读者不需要同意本书的每个判断，但希望读完之后能形成一种更稳固的工程直觉：当一个智能体系统失败时，不要只问“模型为什么没有听懂”，也要问“harness 是否给了它正确的上下文、正确的工具、正确的边界、正确的反馈和正确的停止条件”。当一个智能体系统看起来可用时，也不要只问“它今天能不能跑通 demo”，还要问“它能否被审计、被回滚、被评测、被升级，并在组织中长期维护”。

Harness engineering 的核心，是让模型能力进入工程世界之后仍然可被信任，而不只是让模型看起来更聪明。
